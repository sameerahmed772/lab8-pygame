Project overview

This project is a real-time Pygame simulation where squares chase or flee based on size, age out with per-entity life spans, and are reborn to keep the system alive. The main loop follows a classic order: process input, update world physics, render frame, and cap frame rate.

Mechanics explained for students

Chase: bigger square adds acceleration toward smaller nearby targets, scaled by distance within chase radius.
Flee: smaller square adds acceleration away from bigger nearby threats, scaled by distance within flee radius.
Lifespan and rebirth: each square stores birth time and random life span; once expired it is removed and replaced.
Collisions: if two rectangles overlap, their velocity vectors are swapped for a simple bounce feel.
Input controls: P toggles pause, Space spawns a square, 1 2 3 change difficulty, left click removes one square and adds score.

Key constants and behavior effect

FLEE_RADIUS 150
Small squares start evasion only when a bigger square enters this radius.
CHASE_RADIUS 200
Big squares can pressure targets from farther away than flee trigger range.
FLEE_FORCE 0.8
Higher value means stronger evasive turns and less direct collisions.
CHASE_FORCE 0.5
Higher value means predators lock on faster and close distance earlier.
FRICTION 0.995
Slight damping smooths movement and avoids endless speed growth.
SPEED_BASE_CONSTANT 100
Max speed is inverse of size, so smaller squares feel more agile.
ACCELERATION_RANGE
Random jitter keeps trajectories from becoming too deterministic.
MIN and MAX LIFE SPAN
Controls how fast population refreshes and visual variety over time.
Tuning tip: adjust one force and one radius at a time, then watch whether motion becomes noisy or coherent.

Potential Issues
⚠️Quadratic pair checks each frame
Chase and collision logic both compare many square pairs every update, which can rise quickly as square count grows.
Basics:More squares means much more work each frame.
Engineering Insight:Current complexity is roughly O(n^2) in update paths.
Architecture Insight:Spatial partitioning can reduce pair checks to local neighbors.
Try uniform grid buckets so each square checks only nearby cells.

Tests import names and call signatures that do not exist in current main implementation, so confidence can be misleading.
Basics:If tests cannot run, passing status is not trustworthy.
Engineering Insight:Broken contracts between tests and source hide regressions.

Type hint strengths

  • Function parameters are generally annotated in main functions.
  • Return types are present for key functions and side effect functions use None.
  • TypedDict gives clear field contracts for square records and game state.
  • Container generics such as List, Dict, Tuple are used consistently in main module.

Type hint gaps

  • Test code references symbols and function signatures not present in main module.
  • Mutable dictionary based entities are still prone to key typo mistakes at runtime.
Medium term option: migrate Square to dataclass for stronger editor support and safer refactors.

Surface prebuild for grid background
Background grid is drawn once into an offscreen surface and reused every frame.
Size based speed cap
Velocity limiting prevents unstable accelerations that could trigger expensive corrective behavior.
Performance risks
Two pairwise loops in update
Steering pass and collision pass both scale with square count squared and dominate CPU first.
Frequent random and hypot calls
Per pair calculations use random and distance math heavily; this can become measurable at larger populations.
